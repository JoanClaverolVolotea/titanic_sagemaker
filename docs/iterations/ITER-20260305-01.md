# ITER-20260305-01

## Objetivo y contexto
Agregar una copia local del repo oficial `aws/sagemaker-python-sdk` dentro de `titanic_sagemaker` para consulta tecnica y alineamiento con la libreria oficial, evitando publicar esa copia en el remoto privado.

## Decisiones tecnicas y alternativas descartadas
1. Se mantiene el SDK en `vendor/sagemaker-python-sdk/` como referencia **local-only**.
2. Se ignora el vendor en Git con `.gitignore` (`vendor/sagemaker-python-sdk/`) para evitar que se suba al remoto.
3. Se creo `scripts/update-sdk.sh` para refrescar el vendor desde upstream usando `git clone --depth 1` + `rsync --delete` (sin versionar el vendor en este repo).
4. Alternativas descartadas para el branch principal:
   - `git subtree` persistente: deja historial del SDK en commits y se publicaria al hacer push.
   - `git submodule`: no era la estrategia elegida para este repositorio.

## IAM usado (roles/policies/permisos clave)
1. No aplica IAM/AWS en esta iteracion.
2. No se ejecutaron comandos AWS CLI ni Terraform.

## Comandos ejecutados y resultado esperado
1. Ignorar vendor del SDK:
```bash
echo "vendor/sagemaker-python-sdk/" >> .gitignore
```
Esperado: Git no trackea cambios del SDK local.

2. Refrescar SDK local desde upstream:
```bash
scripts/update-sdk.sh --branch master
```
Esperado: `vendor/sagemaker-python-sdk/` actualizado con upstream, sin cambios staged en Git.

3. Verificacion rapida de estado:
```bash
git status --short
```
Esperado: no aparecen archivos de `vendor/sagemaker-python-sdk/`.

## Evidencia
1. Regla agregada en `.gitignore`: `vendor/sagemaker-python-sdk/`.
2. Script creado: `scripts/update-sdk.sh`.
3. Iteracion registrada: `docs/iterations/ITER-20260305-01.md`.

## Riesgos/pendientes
1. Si alguien usa `git add -f vendor/sagemaker-python-sdk`, podria incluirse accidentalmente el vendor.
2. Cambios mayores de estructura en upstream pueden requerir ajustes del script de sincronizacion.
3. Se recomienda validar periodicamente que `git status` no muestre archivos del vendor.

## Proximo paso
1. Ejecutar `scripts/update-sdk.sh --branch master` al inicio de cada iteracion tecnica que dependa del SDK.
2. Documentar en PR checklist una verificacion explicita de no incluir `vendor/sagemaker-python-sdk/` en commits.
3. Revisar wrappers internos para mapear cambios API contra el SDK oficial en cada update.
