# ITER-20260305-01

## Objetivo y contexto
Agregar el repo oficial `aws/sagemaker-python-sdk` como referencia local dentro de `titanic_sagemaker`, con capacidad de actualizar desde upstream sin subir esa copia al remoto privado del proyecto.

## Decisiones tecnicas y alternativas descartadas
1. Se uso `git subtree` (modo `--squash`) para importar el contenido completo del SDK en `vendor/sagemaker-python-sdk`.
2. Se agrego `vendor/sagemaker-python-sdk/` a `.gitignore` y luego se quito del index con `git rm --cached` para mantener la copia solo local.
3. Se agrego script `scripts/update-sdk.sh` para automatizar el flujo de update: retrack temporal -> subtree pull -> untrack.
4. Alternativas descartadas:
   - `git submodule`: separacion limpia, pero no era la estrategia elegida para este repo.
   - clone manual + copy: mas simple, pero sin trazabilidad integrada de merges tipo subtree.

## IAM usado (roles/policies/permisos clave)
1. No aplica IAM/AWS en esta iteracion.
2. No se ejecutaron comandos AWS CLI ni Terraform.

## Comandos ejecutados y resultado esperado
1. Configurar remoto upstream del SDK:
```bash
git remote add sagemaker-sdk https://github.com/aws/sagemaker-python-sdk.git
git fetch sagemaker-sdk
```
Esperado: remoto agregado y ramas/tags disponibles.

2. Importar SDK completo con subtree:
```bash
git subtree add --prefix=vendor/sagemaker-python-sdk sagemaker-sdk/master --squash
```
Esperado: directorio `vendor/sagemaker-python-sdk` creado con contenido oficial.

3. Evitar publicacion del vendor en remoto:
```bash
git rm -r --cached vendor/sagemaker-python-sdk
```
Esperado: archivos del vendor salen del index, permanecen en disco local.

4. Script de mantenimiento:
```bash
scripts/update-sdk.sh
```
Esperado: actualiza desde upstream y deja vendor nuevamente fuera del index.

## Evidencia
1. Remoto local agregado: `sagemaker-sdk`.
2. Directorio local presente: `vendor/sagemaker-python-sdk/`.
3. Regla de exclusión agregada: `.gitignore` incluye `vendor/sagemaker-python-sdk/`.
4. Script creado: `scripts/update-sdk.sh`.

## Riesgos/pendientes
1. El flujo de update con subtree requiere commits temporales; si el repo esta sucio el script falla por seguridad (a menos que se use `--allow-dirty`).
2. Si upstream cambia branch por defecto, se debe ejecutar con `--branch <nuevo-branch>` o ajustar `UPSTREAM_BRANCH`.
3. Cualquier `git add -f vendor/sagemaker-python-sdk` manual puede volver a incluir archivos si no se sigue el flujo recomendado.
